## Вопросы и ответы

1. Понимание паттерна Page Object:

   **Ответ:** Page Object представляет страницу или значимую область UI как объект с понятными действиями и состояниями. Он скрывает locators и Selenium details от test, поэтому изменения UI обычно исправляются в одном месте. Page Object не должен содержать assertions бизнес-сценария, кроме проверок собственного состояния или invariant, принятых командой.

2. понимать концепцию, уметь создавать простые классы страниц и выносить локаторы

   **Ответ:** Класс страницы хранит locators как private fields и предоставляет методы вроде `loginAs()` или `openOrder()`, которые выражают действия пользователя. Driver передаётся через constructor или controlled factory, а elements ищутся в момент действия, чтобы снизить риск stale references. Test использует публичный API страницы и не обращается к locators напрямую.

3. Понимание 3-уровневой архитектуры:

   **Ответ:** Три уровня разделяют test scenarios, reusable business steps и technical interaction layer. Test отвечает за цель и assertions, Step Layer объединяет операции предметной области, а Page/API Layer работает с UI или service. Границы уменьшают дублирование, но каждый дополнительный уровень должен иметь отдельную ответственность.

4. разделение на «тест → шаг → страница»

   **Ответ:** Test вызывает высокоуровневый шаг, например оформление заказа, Step выполняет последовательность domain actions, а Page Objects управляют конкретными elements. Test остаётся коротким и читаемым, Page не знает о полном сценарии, а Step не хранит locators. Для простой операции лишний Step method может только дублировать Page API, поэтому abstraction вводят осознанно.

5. MVC

   **Ответ:** MVC разделяет application на Model, View и Controller и прежде всего описывает архитектуру продукта, а не готовую структуру test framework. Знание MVC помогает выбирать test levels: Model и business logic проверять ниже, Controller — через API/component tests, View — через UI. Не стоит искусственно называть Page Object «View», а test — «Controller», если обязанности не соответствуют паттерну.

6. Критерии хорошего автотеста

   **Ответ:** Хороший автотест проверяет один понятный риск, имеет однозначный ожидаемый результат и стабильно воспроизводится. Он независим, управляет своими данными, быстро даёт обратную связь и при падении показывает причину через assertion и artifacts. Test также должен быть проще и дешевле в поддержке, чем ценность обнаруживаемого им дефекта.

7. Конфигурация:

   **Ответ:** Конфигурация отделяет код от environment-specific значений: URL, browser, timeouts, credentials и feature flags. Значения читаются из typed model с понятными defaults и validation. Secrets хранятся в environment variables или secret manager, а не в repository.

8. основы запуска: setUp(), tearDown(), конфигурация тестов

   **Ответ:** `setUp()` подготавливает только необходимое состояние: driver/client, test data и начальный context. `tearDown()` закрывает resources и очищает данные даже после failed test, обычно через lifecycle hooks framework. Общая тяжёлая подготовка ускоряет запуск, но создаёт зависимость tests, поэтому scope fixture выбирают осторожно.

9. Архитектура и масштабируемость:

   **Ответ:** Масштабируемая архитектура позволяет добавлять tests, platforms и contributors без роста связности и времени диагностики. Для этого нужны modules с явными dependencies, единая configuration, изоляция state, parallel-safe clients и стабильные extension points. Масштабируемость измеряется не количеством patterns, а стоимостью изменения и запуска.

10. опыт в написании фреймворка базирующегося на уровневой архитектуре (3 уровня)

   **Ответ:** Здесь нужен реальный пример: «В `[учебном или рабочем проекте]` я разделил framework на tests, business steps и pages/API clients. Моей задачей было `[ваша работа]`, а разделение решило `[реальная проблема]`». Если коммерческого опыта нет, следует честно показать учебный repository и объяснить зависимости между слоями.

11. понимание 4-ех и 5-ти уровневой архитектуры

   **Ответ:** Дополнительными уровнями могут быть Component Layer и Element Layer для UI либо Domain Layer и Transport Layer для API. Например: Test → Step → Page → Component → Element wrapper. Пять уровней не лучше трёх автоматически: если слой только перенаправляет вызов без собственного смысла, он усложняет navigation и debugging.

12. Element Layer

   **Ответ:** Element Layer предоставляет общие операции над базовыми controls: click с ожиданием, ввод, получение состояния, логирование и artifacts. Он может реализовывать Button, Input или Select wrappers поверх WebElement и locator. Layer не должен скрывать ошибки бесконечными retries или превращать любой element в универсальный объект с сотней методов.

13. Component Layer

   **Ответ:** Component Object описывает повторяемую часть UI, например header, table, modal или date picker. Он имеет root locator и ищет внутренние elements относительно него, а Page собирается из components. Это уменьшает дублирование и отражает component architecture UI лучше, чем один большой Page class.

14. деление тестов на E2E / UI / API / Integration внутри фреймворка

   **Ответ:** Tests разделяют по modules, source sets, packages и tags, чтобы независимо управлять dependencies и pipeline stages. Общими могут быть domain models, data builders и reporting, но UI driver не должен становиться dependency чистых API tests. Название уровня фиксирует фактические границы, а не только используемый инструмент.

15. Понимание паттерна Page Element.

   **Ответ:** Page Element, или Component Object, представляет отдельный сложный элемент страницы с собственными locators и поведением. Например, таблица умеет найти строку по id и открыть action menu, но не знает полный business scenario. Page Object использует такие components через composition.

16. Поддержка разных типов тестов:

   **Ответ:** Общий framework может поддерживать UI, API, contract, messaging и mobile tests через отдельные adapters и lifecycle. Shared core содержит только действительно общие models, configuration и utilities. Специфичные dependencies изолируются, чтобы изменение browser library не ломало API module.

17. сколько уровней нужно для тестов в Gherkin стиле

   **Ответ:** Минимально нужны Feature и тонкие Step Definitions, а технические действия лучше делегировать Domain Steps и Page/API Layer. Часто получается цепочка Feature → Step Definition → Step/Domain Layer → Page или Client, но фиксированного числа нет. Главное — не писать Selenium или HTTP implementation прямо в Gherkin glue и не добавлять пустые forwarding layers.

18. поддержка Gherkin-стиля + обычных тестов в рамках одного фреймворка (например, JUnit + Cucumber)

   **Ответ:** JUnit tests и Cucumber scenarios могут использовать одни Page Objects, API clients, data builders и assertions. Runner и lifecycle остаются отдельными: Cucumber state изолируется на Scenario, JUnit fixtures — по его scopes. Общий core не должен зависеть от annotations конкретного runner, иначе reuse станет формальным.

19. разделение Feature → Scenario → Step Definition → Step Layer → Page Layer

   **Ответ:** Feature группирует business capability, Scenario показывает конкретное поведение, а Step Definition связывает текст с кодом. Step Layer выполняет reusable domain operation, Page Layer взаимодействует с UI. Assertions могут находиться в domain verification service или тонком Then step, но Page не должен знать Gherkin text.

20. Конфигурируемость и адаптивность:

   **Ответ:** Framework должен менять environment, browser, tags, parallelism и integrations без перекомпиляции test code. Configuration валидируется до запуска и выводится в отчёт без secrets. Адаптивность не означает поддержку любой комбинации: допустимые варианты и defaults должны быть ограничены.

21. гибкое управление конфигурацией запуска

   **Ответ:** Удобна явная иерархия: command-line или CI variables переопределяют environment variables, затем configuration file и безопасные defaults. Значения преобразуются в immutable typed object и проверяются до создания resources. Это предотвращает ситуацию, когда разные layers читают одну переменную по-разному.

22. локализация и мультиязычность UI

   **Ответ:** Locale задаётся в browser/device и test data, а expected translations читаются из контролируемого resource или API contract. Locators должны опираться на stable attributes, а не на переведённый текст, кроме tests самой локализации. Проверяются encoding, layout, форматы дат/чисел, plural forms, направление текста и fallback language.

23. Опыт в применении различных уровневых архитектур, как для UI так и для API:

   **Ответ:** Здесь нужен честный сравнительный пример: `[какая архитектура использовалась для UI]` и `[какая для API]`, какие layers добавили и почему. UI обычно требует Pages/Components/Elements, а API — Domain Services/Clients/Serialization. Если опыт учебный, это следует прямо указать и показать осознанный выбор, а не выдумывать production framework.

24. понимание и применение: Hexagonal, Clean Architecture, CQRS, Ports and Adapters

   **Ответ:** Hexagonal и Ports and Adapters отделяют domain от внешних systems через ports и adapters; Clean Architecture также направляет dependencies к business rules. CQRS разделяет command и query models, что влияет на test data и проверки eventual consistency. В test framework эти идеи полезны для замены UI/API/mocks adapters, но не требуют копировать всю архитектуру продукта.

25. выделение domain-слоя в API/contract-тестировании

   **Ответ:** Domain Layer описывает операции вроде `createOrder` и business models, не зная деталей HTTP library. Transport adapter формирует request, сериализует body и возвращает response, а contract layer проверяет schemas и interactions. Такое разделение позволяет переиспользовать domain scenarios для real client и controlled stub.

26. декомпозиция UI-тестов по BFF/SPA/SSR

   **Ответ:** В SPA значительная логика находится в browser, поэтому важны component tests, API mocking и небольшой E2E layer. При SSR проверяется server-rendered HTML, hydration и последующие client interactions, а BFF отдельно покрывается API/contract tests по needs конкретного frontend. UI tests декомпозируют по ответственности, а не дублируют всю цепочку в каждом scenario.

27. Архитектура базирующаяся на проекте девелоперов (плюсы, минусы):

   **Ответ:** Плюс — tests используют те же domain terms, modules и contracts, поэтому проще понимать impact changes и переиспользовать models. Минус — сильная связь с implementation может привести к одинаковым ошибкам и дорогим изменениям test code вместе с product code. Я сохраняю независимые assertions и public contracts, заимствуя structure только там, где она отражает устойчивые границы.

28. реализация архитектуры автотестов, соответствующей архитектуре delivery pipeline:

   **Ответ:** Modules framework сопоставляются pipeline stages: Unit/contract tests выполняются до build artifact, component tests — с isolated dependencies, Smoke — после deployment, E2E — на готовом environment. Каждый module имеет собственную команду, tags и artifacts. Это даёт быстрый fail и не заставляет ждать UI suite для ошибки compile или contract.

29. CI/CD, Kubernetes, Helm, cloud services (AWS, GCP)

   **Ответ:** CI собирает test artifact или container и запускает выбранные modules, Helm/Kubernetes могут развернуть временное environment, а cloud services предоставляют databases, queues и observability. Framework получает endpoints и credentials после deployment и публикует correlation data. Tests не должны иметь постоянные administrator credentials или предполагать конкретный cloud API без adapter.

30. учет особенностей платформ: Web, Mobile, Backend, Messaging (Kafka, RabbitMQ)

   **Ответ:** Web использует browser contexts и DOM, Mobile — devices, permissions и app lifecycle, Backend — HTTP/RPC и data stores, Messaging — asynchronous delivery и broker semantics. Общими остаются domain models, configuration и reporting, а drivers и waits выделяются в platform adapters. Одинаковый abstraction для click и Kafka event только скрывает различия.

31. Архитектура основанная на компонентах:

   **Ответ:** Component-based framework собирается из modules с небольшими public interfaces и явными dependencies. Компонент можно независимо тестировать, версионировать и подключать только нужным suites. Shared module не должен превращаться в место для любой случайной utility.

32. core + modules + shared libs, переиспользуемые компоненты

   **Ответ:** Core содержит contracts, configuration interfaces и lifecycle abstractions, modules реализуют UI, API, messaging или reporting, а shared libraries дают стабильные data builders и domain models. Dependency direction идёт от modules к небольшому core. Публикация shared library оправдана, только если есть несколько реальных consumers и versioning policy.

33. Микросервисная архитектура

   **Ответ:** Для каждого service нужны Unit/component tests, consumer/provider contracts и ограниченные integration scenarios с настоящими dependencies. E2E tests покрывают только критичные cross-service flows, потому что они медленные и сложные в диагностике. Framework группирует tests по service ownership и позволяет запускать impacted scope независимо.

34. Арxитектура для Contract testing

   **Ответ:** Consumer tests формируют ожидания contract, broker или repository хранит версии, а provider verification запускается против provider build. Test data и provider states задаются явно, результаты публикуются для решения о совместимости deployment. Contract module отделяется от E2E и не требует полного общего environment.

35. Возможность масштабирования архитектуры:

   **Ответ:** Для масштабирования нужны stateless runners, isolated data, thread-safe resources, разделение suites и возможность горизонтально добавлять agents. Execution time, queue и flaky rate измеряются до и после изменений. Простое увеличение числа machines не поможет, если tests зависят от одного shared account или environment bottleneck.

36. параллельные прогоны

   **Ответ:** Каждый поток или process получает независимый driver/client context, test data и artifacts path. Shared caches должны быть immutable или synchronized, а cleanup — привязан к owner test. Уровень параллелизма ограничивается capacity приложения, Grid, database и внешних services, иначе tests сами создают искусственные failures.

37. изолированная конфигурация сред

   **Ответ:** Environment profile содержит endpoints, capabilities и non-secret defaults, а secrets поступают отдельно. Configuration object создаётся на test run или worker и не изменяется глобально. Tests явно отмечают разрешённые environments, чтобы destructive scenario не попал в production.

38. динамическое создание/уничтожение окружений (e.g., test containers)

   **Ответ:** Testcontainers или аналогичный инструмент поднимает versioned dependency, например database или broker, перед suite и удаляет её после выполнения. Tests получают динамический port и credentials из созданного resource, а readiness определяется health check, не sleep. Container повышает воспроизводимость, но не полностью повторяет managed production service и требует ресурсов CI.
