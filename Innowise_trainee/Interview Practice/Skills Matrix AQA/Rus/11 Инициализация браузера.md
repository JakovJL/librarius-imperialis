## Вопросы и ответы

1. Что такое selenium webdriver?

   **Ответ:** Selenium WebDriver — это API для управления браузером через стандарт WebDriver. Тестовый код отправляет команды browser driver или удалённому Selenium server, а браузер выполняет их почти так же, как действия пользователя. WebDriver отвечает за автоматизацию браузера, но assertions, test lifecycle и отчётность обычно предоставляет test framework.

2. Где прописать путь к драйверам?

   **Ответ:** В актуальном Selenium обычно не нужно указывать путь вручную: Selenium Manager определяет браузер, загружает совместимый driver и использует cache. При ручной настройке binary driver можно добавить в системный `PATH` или передать путь через browser-specific `Service`; старый вариант для Java — system property вроде `webdriver.chrome.driver`. В CI версии браузера и driver лучше контролировать, чтобы сборка оставалась воспроизводимой.

3. Реализация инициализации webdriver.

   **Ответ:** Сначала создаются browser options, например headless mode, размер окна и capabilities, затем на их основе создаётся `ChromeDriver`, `FirefoxDriver` или `RemoteWebDriver`. После создания я настраиваю timeouts и открываю нужный URL. Driver обязательно закрывается через `quit()` в teardown или `finally`, даже если тест упал.

4. Браузерные движки(gecko, webKit, edge)

   **Ответ:** Gecko — browser engine Firefox, WebKit используется Safari, а Chromium-браузеры Chrome и современный Edge используют Blink. Edge не является названием отдельного движка: прежний EdgeHTML больше не используется в актуальном Chromium-based Edge. Selenium взаимодействует с браузерами через соответствующие drivers, например GeckoDriver, SafariDriver и EdgeDriver.

5. Реализация инициализации singleton, абстрактной фабрики.

   **Ответ:** Singleton возвращает один WebDriver instance на всё приложение, но обычный глобальный Singleton создаёт проблемы при parallel execution и связывает тесты общим состоянием. Abstract Factory лучше отделяет выбор браузера: общий интерфейс создаёт driver, а реализации настраивают Chrome, Firefox или RemoteWebDriver. Для параллельных тестов я бы сочетал factory с driver lifecycle на тест или с `ThreadLocal`, а не использовал один browser session для всех потоков.

6. Browser capabilities.

   **Ответ:** Capabilities — это параметры новой browser session, например `browserName`, `browserVersion`, `platformName`, proxy и browser-specific options. В Selenium 4 их обычно задают через `ChromeOptions`, `FirefoxOptions` или другой класс Options и передают при создании driver. Remote end согласует запрос с доступной конфигурацией, поэтому фактические capabilities session полезно проверять после запуска.

7. Browser multiCapabilities.

   **Ответ:** Одна WebDriver session в итоге получает один согласованный набор capabilities. На уровне W3C new session request может содержать общие `alwaysMatch` и альтернативы `firstMatch`, из которых remote end выбирает совместимый вариант. Для полноценной матрицы браузеров и платформ обычно создают отдельную session на каждую комбинацию через параметры тестов или CI, а не пытаются управлять несколькими браузерами одним driver.

8. Remote webdriver.

   **Ответ:** `RemoteWebDriver` выполняет тестовый код на клиенте, но отправляет команды по URL удалённому Selenium Grid или cloud service, где запускается браузер. При создании передаются Grid URL и Options с нужными capabilities. Это удобно для parallel и cross-browser testing, но требует управления сетью, доступностью nodes, версиями браузеров, очередью sessions и artifacts.

9. Headless mode.

   **Ответ:** В headless mode браузер работает без видимого окна, что удобно для CI и серверов. Режим включается browser option, например аргументом `--headless`, а размер viewport лучше задавать явно. Headless не делает плохой тест стабильным автоматически и по-прежнему использует реальный browser engine.

10. Реализация своей обертки над элементами.

   **Ответ:** Обёртка инкапсулирует повторяющиеся операции над WebElement: ожидание состояния, click, ввод, логирование и добавление artifacts при ошибке. Я предпочитаю composition: обёртка хранит locator и driver или найденный элемент, а не наследуется от внутренних реализаций Selenium. API должен оставаться небольшим и не скрывать реальные причины ошибок за автоматическими повторными попытками.

11. Реализация и инициализации multiton или prototype.

   **Ответ:** Multiton хранит несколько instances по ключу, например отдельный WebDriver для каждого thread или browser type. Prototype создаёт независимый новый объект на основе общей конфигурации, поэтому хорошо подходит для изолированной session на каждый тест. На практике driver factory с явным lifecycle или `ThreadLocal<WebDriver>` обычно понятнее, а все созданные sessions нужно гарантированно закрывать.

12. Подводные камни headless режима.

   **Ответ:** Headless может отличаться размером viewport, доступными fonts, GPU settings, download behavior и обработкой системных dialogs. Из-за отличий контейнера или CI могут измениться layout, screenshots и timing, хотя причина не всегда находится именно в headless. Я фиксирую browser version и window size, устанавливаю нужные fonts и при непонятном падении воспроизвожу тест в headed mode на том же окружении.

13. Selenium логирование.

   **Ответ:** Полезно разделять логи тестового framework, browser driver, Selenium Grid и самого браузера. Driver service и Grid можно запускать с нужным log level и сохранять их output как CI artifacts; browser console и network events получают через поддерживаемые logging APIs или WebDriver BiDi. В логах должны быть session id, browser capabilities и шаг теста, но secrets и персональные данные нужно маскировать.
