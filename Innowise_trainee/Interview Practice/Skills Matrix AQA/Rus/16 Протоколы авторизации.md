## Вопросы и ответы

1. Базовые понятия:

   **Ответ:** В identity and access management важно разделять подтверждение личности, выдачу прав и передачу tokens. Protocol описывает участников, endpoints, messages и проверки, а конкретный product реализует эти правила. Безопасность зависит не только от выбранного protocol, но и от redirect URI, TLS, key management, scopes и session policy.

2. различие между аутентификацией и авторизацией

   **Ответ:** Authentication отвечает на вопрос «кто пользователь или client», а authorization — «что ему разрешено». Сначала система обычно подтверждает identity, затем проверяет roles, scopes, attributes и ownership ресурса для каждого действия. Успешный login не означает право доступа ко всем endpoints или чужим данным.

3. что такое OpenID.

   **Ответ:** OpenID 1.0/2.0 — более ранние protocols федеративной authentication, позволявшие сайту доверять внешнему identity provider. В современных системах обычно используется OpenID Connect, который добавляет identity layer поверх OAuth 2.0 и выдаёт ID Token. Поэтому на собеседовании важно уточнить, имеется ли в виду старый OpenID или OIDC.

4. что такое OAuth.

   **Ответ:** OAuth — authorization framework для делегированного доступа: client получает ограниченное право обращаться к resource server от имени пользователя или самого client. Пароль пользователя не передаётся стороннему приложению. OAuth сам по себе не является protocol пользовательской authentication; для этого обычно используют OpenID Connect.

5. что такое OAuth 2.0

   **Ответ:** OAuth 2.0 — стандартизированный framework, в котором authorization server выдаёт client access token для доступа к protected resources. Flow зависит от типа client и сценария, например Authorization Code с PKCE для пользовательского входа или Client Credentials для machine-to-machine. Access token имеет ограниченный scope и lifetime и передаётся resource server через защищённый канал.

6. основные роли OAuth (Resource Owner, Client, Authorization Server, Resource Server)

   **Ответ:** Resource Owner может предоставить доступ к защищённому ресурсу, а Client запрашивает этот доступ. Authorization Server подтверждает grant и выдаёт tokens, Resource Server хранит API и проверяет access token и его права. Один product может физически совмещать несколько ролей, но логически их обязанности остаются разными.

7. Понимание токенов:

   **Ответ:** Token представляет выданные права или identity claims и должен использоваться только по назначению. Проверяются type, issuer, audience, lifetime, scope, signature или introspection status. Token нельзя считать безопасным только потому, что его содержимое выглядит правильным.

8. JWT, access/refresh token, их структура

   **Ответ:** JWT — формат claims; signed JWT обычно состоит из base64url-encoded header, payload и signature, разделённых точками. Access token даёт доступ resource server и может быть JWT или opaque string, а refresh token используется только с authorization server для получения нового access token и тоже не обязан быть JWT. Payload signed JWT не зашифрован, поэтому secret data туда помещать нельзя.

9. API-интеграция:

   **Ответ:** API client должен получать token разрешённым flow, безопасно хранить его в памяти или secret storage и передавать только нужному audience. Integration также обрабатывает expiry, refresh, error responses и logout или revocation policy. Логи и отчёты должны маскировать credentials и tokens.

10. добавление Authorization заголовков в автотесты

   **Ответ:** Для Bearer token используется header `Authorization: Bearer <access_token>`, а для Basic — соответствующая Basic scheme. В framework header лучше добавлять через request filter или authenticated client, а token получать fixture или authorization client. Нельзя коммитить token в код, печатать его полностью или повторно использовать между изолированными tenants без проверки.

11. Глубина понимания:

   **Ответ:** Глубокое понимание означает способность проследить flow от authorization request до проверки resource server и объяснить назначение state, nonce, PKCE, scopes, audience и keys. Также нужно понимать угрозы: token leakage, code interception, redirect attack, replay и excessive privileges. Настройки конкретного provider всегда сопоставляются со стандартом и security policy проекта.

12. различия OAuth, OAuth 2.0

   **Ответ:** Под «OAuth» без версии часто понимают семейство protocols, включая OAuth 1.0 и OAuth 2.0. OAuth 1.0 подписывает каждый request и использует другой flow, а OAuth 2.0 основан на access tokens, TLS и расширяемых grant types; версии не совместимы. OAuth 2.0 — не просто новая minor version, а другая архитектура.

13. Практика:

   **Ответ:** Практика включает регистрацию client, настройку redirect URI, получение tokens, вызов API и проверку refresh, expiry, scopes и logout. Полезно уметь читать discovery metadata и анализировать flow в browser Network без раскрытия secrets. В test environment нужны отдельные clients и users с минимальными правами.

14. иметь опыт работы с одним из протоколов.

   **Ответ:** Здесь нужен честный пример: «Я работал с `[OAuth 2.0, OIDC, SAML или другим protocol]` в `[учебном или рабочем контексте]`. Я настраивал или тестировал `[реальные действия]` и проверял `[flow, roles, tokens или errors]`». Если опыта нет, лучше описать самостоятельно реализованный учебный flow и не выдавать его за коммерческий проект.

15. работа с endpoints (Token, Authorization), запрос токенов программно

   **Ответ:** Authorization Endpoint взаимодействует с user agent, выполняет login/consent и возвращает authorization code на зарегистрированный redirect URI. Client обменивает code и PKCE verifier на Token Endpoint через back-channel; machine client может программно использовать Client Credentials, если это разрешено. Автоматически вводить пароль пользователя в legacy password grant не стоит, если provider не требует его для старой системы.

16. Безопасность:

   **Ответ:** Все endpoints используют TLS, redirect URIs сопоставляются строго, tokens имеют минимальные scope и lifetime, а public clients применяют PKCE. Secrets, authorization codes и tokens не попадают в URL logs или artifacts. Authorization Server должен поддерживать актуальные algorithms и key rotation, а Resource Server — проверять каждое обращение.

17. тесты на истекшие, поддельные, некорректные токены

   **Ответ:** Я проверяю отсутствующий token, неправильную scheme, повреждённый формат, изменённый payload или signature, неверные issuer/audience, истёкший `exp` и token с будущим `nbf`. Resource Server должен отклонять их согласованным 401 response и не раскрывать внутренние детали. Недостаточный scope или role обычно приводит к 403, если identity уже подтверждена.

18. Token Endpoint, Authorize Endpoint - Indentity Server.

   **Ответ:** Identity Server или Authorization Server публикует Authorization Endpoint для front-channel interaction и Token Endpoint для выдачи tokens по валидному grant. В OIDC адреса обычно обнаруживаются через discovery document, а не прописываются догадкой. Token Endpoint требует TLS и подходящую client authentication или PKCE validation в зависимости от типа client.

19. Протоколы:

   **Ответ:** OAuth 2.0 отвечает за delegated authorization, OIDC добавляет authentication и identity claims, SAML передаёт XML assertions для federation, а JWT/JWK являются formats, а не самостоятельными login protocols. PKCE — защита authorization code flow. Выбор зависит от типа clients, экосистемы и требований организации.

20. углубленное знание OAuth 2.0/2.1, OIDC, SAML, JWT, PKCE, JWK

   **Ответ:** OAuth 2.0 — действующий framework, а OAuth 2.1 на текущий момент остаётся Internet-Draft, который объединяет актуальные практики и исключает устаревшие flows. OIDC использует OAuth 2.0 и ID Token для authentication, SAML работает с XML assertions, JWT хранит claims, JWK описывает cryptographic key, а PKCE связывает authorization request и token exchange через verifier. Эти понятия дополняют друг друга, но не взаимозаменяемы.

21. понимание работы Oauth 2.0(refresh token, access token, roles,..)

   **Ответ:** После валидного grant Authorization Server выдаёт короткоживущий access token и при необходимости refresh token. Client отправляет access token Resource Server, который проверяет validity, audience, scopes и применяет собственные roles и object-level rules. Refresh token отправляется только Token Endpoint, защищается строже и может rotation-иться; это уменьшает необходимость долгоживущего access token.

22. опыт работы с двумя и более протоколами авторизации  Oauth 2.0

   **Ответ:** Ответ должен содержать только реальный опыт: «Я использовал `[protocol 1]` для `[цель]` и `[protocol 2]` для `[цель]`; моей задачей было `[конкретные действия]`». Следует сравнить flows, token/assertion format и способ integration, а не просто перечислить названия. Если опыта только с OAuth 2.0, лучше сказать об этом прямо и объяснить понимание OIDC или SAML теоретически.

23. Authorize Endpoint - Indentity Server

   **Ответ:** Authorization Endpoint принимает authorization request с `client_id`, `redirect_uri`, `response_type`, `scope`, `state` и при OIDC дополнительными parameters. User agent проходит authentication и consent, после чего server возвращает response на строго зарегистрированный redirect URI. `state` проверяется client для связывания request и защиты от CSRF, а OIDC `nonce` связывает authentication request с ID Token.

24. Инфраструктура:

   **Ответ:** Identity infrastructure включает Authorization Server, user directory, clients, Resource Servers, keys, certificates, session storage и audit systems. Configuration должна версионироваться, secrets храниться отдельно, а discovery и JWKS endpoints быть доступными и наблюдаемыми. Резервирование и key rotation проверяются так же, как функциональные flows.

25. Keycloak/Auth0/Azure AD, mock-authorization server

   **Ответ:** Keycloak — self-hosted identity and access management product, Auth0 — managed identity platform, а Azure AD сейчас называется Microsoft Entra ID. Все могут предоставлять OAuth 2.0/OIDC и enterprise integrations, но имеют разные configuration models. Mock authorization server полезен для controlled tokens и errors в component tests, однако реальные integration tests нужны для redirect, claims, policies и key rotation.

26. Безопасность:

   **Ответ:** Основные принципы — Authorization Code с PKCE для пользовательских public clients, строгие redirect URIs, TLS, минимальные scopes, короткий lifetime и защищённое хранение refresh tokens. Проверяются algorithm allowlist, issuer, audience, expiry, nonce/state и rotation keys. Client не должен принимать token только после base64 decoding без cryptographic validation.

27. тесты на подделку токенов, отсутствие scope/role, introspection

   **Ответ:** Для signed token изменяю header, payload или signature, использую неизвестный `kid`, запрещённый algorithm, неверные issuer и audience и убеждаюсь, что доступ отклонён. Отдельно проверяю валидный token без нужного scope/role и попытку доступа к чужому object. Для opaque token Resource Server может использовать introspection; tests покрывают active=false, expiry, неправильный client и недоступность endpoint согласно fail-closed policy.

28. Интеграция:

   **Ответ:** Integration связывает discovery, authorization flow, callback, token validation, user session и authorization внутри API. Контракты claims и scopes должны быть согласованы между Identity Server, clients и Resource Servers. Изменения configuration сначала проверяются в отдельном environment и разворачиваются с возможностью rollback.

29. сложные CI/CD сценарии, multi-tenant support

   **Ответ:** Pipeline создаёт изолированные clients/users или использует подготовленные fixtures для каждого tenant, получает tokens и запускает матрицу positive и negative access tests. Secrets берутся из vault, а logs маскируются; параллельные tests не должны менять общую identity configuration. Для multi-tenant проверяются issuer, audience, tenant claim, key set и запрет использования token одного tenant в другом.

30. Мониторинг:

   **Ответ:** Мониторинг отслеживает availability и latency endpoints, ошибки login/token exchange, аномалии refresh, rejected tokens и подозрительные authorization attempts. Alerts строятся по baseline и business impact, чтобы не создавать шум. Метрики не должны содержать raw tokens, passwords или лишние персональные данные.

31. аудит логов, мониторинг токенов, визуализация сессий

   **Ответ:** Audit log фиксирует время, actor, client, tenant, действие, результат, reason code и correlation id, но не полный token. Для tokens отслеживаются issue, refresh, revoke, expiry и validation failures по безопасным identifiers. Визуализация sessions помогает находить необычную длительность, множество devices или массовые ошибки, при этом доступ к таким данным ограничивается и соответствует retention policy.
