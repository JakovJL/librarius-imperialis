## Вопросы и ответы

1. Понимание разницы между виртуализацией (VMs) и контейнеризацией (e.g., Docker).-

   **Ответ:** VM виртуализирует hardware и запускает отдельную guest OS поверх hypervisor, поэтому даёт сильную изоляцию, но требует больше ресурсов. Container изолирует process и его filesystem, используя kernel host OS, поэтому запускается быстрее и имеет меньший overhead. Container не является облегчённой VM, а уровень изоляции и совместимость kernel у этих подходов различаются.

2. Объяснить, для чего необходима виртуализация и контейнеризация в процессе разработки и тестирования ПО

   **Ответ:** Они позволяют создавать повторяемые и изолированные environments без отдельного физического server для каждого проекта. VM подходит, когда нужна другая OS или сильная граница, container — для быстрого запуска application и dependencies. В тестировании это упрощает parallel runs, воспроизведение дефектов и очистку состояния.

3. Какие проблемы они помогают решить (e.g., "works on my machine" problem, environment consistency).

   **Ответ:** Versioned image фиксирует runtime, libraries и системные dependencies, поэтому local и CI environments становятся ближе. Изоляция уменьшает конфликты ports и versions между проектами, а environment можно пересоздать вместо ручного ремонта. Однако внешние services, hardware и configuration всё равно могут отличаться, поэтому container не гарантирует полную идентичность production.

4. Польза контейнеризации (Docker) для запуска автотестов и создания изолированных тестовых окружений.

   **Ответ:** Tests можно упаковать вместе с runtime или запускать рядом с containerized database, broker и browser. Каждый run получает известные versions и собственную network, а после завершения resources удаляются. Это повышает воспроизводимость и parallelism, но требует readiness checks, resource limits и сохранения artifacts до удаления containers.

5. Базовое понимание, что такое Docker image и Docker container.

   **Ответ:** Docker image — immutable package с filesystem layers, binaries, libraries и metadata для запуска. Container — созданный из image изолированный process с собственным writable layer и runtime configuration. Из одного image можно независимо запустить много containers.

6. Знание, что Jenkins agents могут запускаться в Docker контейнерах.

   **Ответ:** Jenkins agent можно запускать из image с JRE и нужными build/test tools, подключая его к controller. Pipeline получает чистое воспроизводимое environment, а agent может удаляться после job. Нужно управлять credentials, workspace, cache и доступом к Docker или Kubernetes API по принципу least privilege.

7. Практический опыт работы с Docker: создание и запуск контейнеров, использование Dockerfile.

   **Ответ:** Здесь нужен реальный пример: «Я собирал image командой `[ваша команда]`, запускал container с `[ports, volumes или variables]` и использовал Dockerfile для `[цель]`». Следует описать одну проблему и способ диагностики. Если опыт только учебный, это нужно прямо сказать и не выдавать за production.

8. Объяснить жизненный цикл контейнера Docker (create, run, stop, rm).

   **Ответ:** `docker create` создаёт container из image, но не запускает main process; `docker start` запускает созданный container. `docker run` обычно объединяет create и start, `docker stop` посылает graceful signal и затем принудительно останавливает по timeout, а `docker rm` удаляет остановленный container. Persistent data следует хранить в volume, потому что удаление container убирает его writable layer.

9. Умение писать простые Dockerfile для контейнеризации тестовых окружений или приложений.

   **Ответ:** Простой Dockerfile начинается с подходящего `FROM`, задаёт `WORKDIR`, копирует dependency manifests, устанавливает dependencies, затем копирует code и задаёт `ENTRYPOINT` или `CMD`. Порядок помогает использовать build cache. Image лучше запускать не от root, исключать лишние files через `.dockerignore` и не помещать secrets в layers.

10. Понимание концепции Docker Hub (или другого registry) для хранения и получения образов.

   **Ответ:** Registry хранит versioned container images, а repository группирует images одного приложения. Docker Hub — публичный и private registry service; альтернативы есть у cloud providers и on-premises products. Pipeline аутентифицируется, публикует tag и digest, а deployment получает проверенный image; mutable tag вроде `latest` не должен быть единственным способом идентификации release.

11. Четкое различие между Docker image и Docker container, включая слои образа (layers).

   **Ответ:** Image состоит из immutable filesystem layers, создаваемых build instructions и переиспользуемых cache. Container добавляет поверх них отдельный writable layer и runtime state, не меняя исходный image. Изменение внутри container исчезнет после его удаления, если данные не вынесены в volume или новый image не собран отдельно.

12. Понимание, что такое base image и как его выбирать.

   **Ответ:** Base image задаётся в `FROM` и предоставляет начальную OS/user space, runtime или минимальный filesystem. Я выбираю официальный или доверенный source, нужную architecture, поддерживаемую version, минимальный набор packages и понятный update policy. Tag или digest фиксируется осознанно, а image регулярно сканируется и пересобирается с security updates.

13. Знание, как настроить и использовать Jenkins agent, запущенный в Docker контейнере, для выполнения тестовых задач.

   **Ответ:** Image agent содержит совместимый JRE и test tools, а Jenkins configuration определяет способ подключения, credentials, labels, volumes и resource limits. Job выбирает label, получает workspace, выполняет tests и публикует artifacts до удаления container. Доступ к Docker socket особенно чувствителен, потому что фактически даёт высокие права на host.

14. Общее понимание, что Ansible может использоваться для конфигурации окружений, где запускаются Docker контейнеры.

   **Ответ:** Ansible через inventory и idempotent modules устанавливает packages, настраивает Docker daemon, networks, users, files и запускает containers на hosts. Playbooks хранятся в version control и уменьшают ручной drift. Secrets передаются через защищённый механизм, а повторный запуск должен безопасно приводить system к declared state.

15. Начальное представление об оркестрации контейнеров (что это и зачем нужно, e.g., Kubernetes на концептуальном уровне)

   **Ответ:** Orchestrator размещает containers на cluster nodes, поддерживает desired state, перезапускает failures, масштабирует replicas и управляет networking и configuration. Он нужен, когда один `docker run` уже не обеспечивает availability и управление множеством workloads. Kubernetes — распространённый orchestrator, но его сложность оправдана не для каждого небольшого environment.

16. Глубокое понимание и практический опыт с системами оркестрации контейнеров (предпочтительно Kubernetes, но можно и Docker Swarm).

   **Ответ:** Этот ответ должен опираться на реальный опыт: `[кластер, workloads, ваша ответственность, проблема и решение]`. Важно объяснить scheduling, rollout, health probes, resources, networking, storage и observability, а не только команды `kubectl`. При отсутствии production опыта можно честно описать local cluster и ограничения такого опыта.

17. Развертывание и управление приложениями в Kubernetes (Pods, Services, Deployments, ConfigMaps, Secrets).

   **Ответ:** Pod содержит один или несколько тесно связанных containers, Deployment управляет replicas и rollout, а Service даёт стабильный network endpoint выбранным Pods. ConfigMap хранит non-secret configuration, Secret — sensitive values, хотя его безопасное хранение и access control требуют дополнительной настройки. Изменения применяются declarative manifests, после чего проверяются rollout status, events и probes.

18. Понимание архитектуры Kubernetes (Master node, Worker nodes, kubelet, kube-proxy, etcd).

   **Ответ:** Актуальный термин вместо Master — control plane: API server принимает requests, etcd хранит cluster state, scheduler выбирает node, а controller managers приводят actual state к desired. Worker node запускает container runtime и kubelet, который поддерживает Pods. `kube-proxy` может реализовывать Service networking, но является optional, потому что некоторые network implementations заменяют его функции.

19. Умение писать Kubernetes manifest-файлы.

   **Ответ:** Manifest в YAML или JSON содержит `apiVersion`, `kind`, `metadata` и `spec`, например для Deployment и Service. Я задаю labels/selectors, image digest или контролируемый tag, ports, environment, resource requests/limits и readiness/liveness probes. Файл проверяется schema validation и policy tools до применения, а secrets не хранятся открытым текстом в Git.

20. Опыт использования Ansible для автоматизации развертывания и конфигурации инфраструктуры под контейнеризированные приложения, включая настройку Kubernetes кластеров или Docker хостов.

   **Ответ:** Нужен честный шаблон: «Я использовал Ansible для `[реальная задача]`, описал inventory, roles и variables, а результат проверял через `[способ]`». Для cluster важно разделять bootstrap hosts и управление Kubernetes resources, для которых часто удобнее declarative manifests или Helm. Если опыта нет, следует объяснить idempotency и показать учебный playbook.

21. Проектирование CI/CD пайплайнов (в Jenkins или TFS/Azure DevOps) для сборки Docker образов, их тестирования и развертывания в Kubernetes.

   **Ответ:** Pipeline проверяет code, собирает image, запускает tests и security scan, публикует immutable tag/digest в registry и обновляет deployment manifest или Helm values. Затем выполняется rollout в environment, Smoke tests и проверка observability; при ошибке останавливается promotion или выполняется согласованный rollback. Credentials выдаются stage-specific identity, а production deployment защищается approvals и policy.

22. Оптимизация Docker образов (размер, безопасность, скорость сборки).

   **Ответ:** Я использую multi-stage build, небольшой trusted base image, `.dockerignore` и копирую dependency manifests до source code для cache. Удаляю build tools из runtime stage, запускаю non-root user, фиксирую versions и сканирую vulnerabilities и secrets. Минимальный размер полезен, но не должен ухудшать debug и поддержку без измеримой причины.

23. Настройка и управление Jenkins agents, работающих в динамически создаваемых контейнерах в Kubernetes (e.g., Jenkins Kubernetes plugin).

   **Ответ:** Jenkins Kubernetes plugin создаёт отдельный agent Pod по template для build и удаляет его после завершения. Template задаёт service account, containers с tools, resources, volumes и pod security settings, а Pipeline выбирает нужный container для steps. Нужно ограничивать права namespace, сохранять artifacts до удаления и контролировать orphaned Pods и cluster capacity.

24. Диагностика и решение проблем в контейнеризированных средах и системах оркестрации.

   **Ответ:** В Docker я проверяю `ps`, logs, inspect, exit code, health, mounts, networks и resource usage. В Kubernetes начинаю с `get`, `describe`, logs и events, затем проверяю probes, image pull, scheduling, limits, DNS, Service endpoints и network policies. Временный `exec` полезен, но исправление должно попасть в image или manifest, а не остаться ручным изменением container.

25. Понимание сетевых аспектов контейнеризации и Kubernetes (e.g., CNI, Services, Ingress).

   **Ответ:** CNI plugin реализует Pod networking и IP connectivity, Service предоставляет стабильный virtual address и load balancing к Pods, а Ingress описывает внешний HTTP/HTTPS routing через установленный controller. NetworkPolicy ограничивает разрешённый traffic, если CNI её поддерживает. Для диагностики проверяются DNS, endpoints, ports, selectors и правила между namespaces.

26. Понимание Docker Host (DOCKER_HOST environment variable or docker -H) и его назначения при работе с удаленным Docker daemon или несколькими Docker инстансами.

   **Ответ:** `DOCKER_HOST` или global flag `docker -H` указывает CLI, к какому daemon socket подключаться, например local Unix socket, SSH или защищённый TCP endpoint. Для нескольких endpoints удобнее Docker contexts, которые хранят endpoint и TLS configuration и переключаются явно. Открытый remote daemon без TLS опасен, потому что доступ к нему может дать контроль над host.

27. Способность оценить и объяснить преимущества и недостатки Kubernetes и Docker Swarm в различных сценариях.

   **Ответ:** Kubernetes имеет богатую ecosystem, гибкое scheduling, controllers, networking и managed cloud offerings, но требует больше знаний и эксплуатации. Docker Swarm проще настроить и понять для небольшого cluster, но ecosystem и возможности управления сложными workloads ограниченнее. Выбор зависит от масштаба, existing platform, компетенций, availability requirements и общей стоимости поддержки.
