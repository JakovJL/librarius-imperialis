## Вопросы и ответы

1. Что такое CI.

   **Ответ:** Continuous Integration — это практика частого объединения небольших изменений в общий репозиторий с автоматической проверкой каждого изменения. Pipeline обычно собирает проект, запускает static analysis и тесты и быстро сообщает результат команде. Цель CI — рано обнаруживать проблемы интеграции и постоянно поддерживать основную ветку в рабочем состоянии.

2. Слышал об инструментах CI (Jenkins, TeamCity, TFS).

   **Ответ:** Jenkins — расширяемый CI-сервер с большим числом плагинов и Pipeline as Code через Jenkinsfile. TeamCity предоставляет готовый интерфейс для build configurations, agents, зависимостей и отчётов. TFS, сейчас Azure DevOps Server, включает репозитории, build и release pipelines, work items и другие сервисы; конкретный инструмент выбирают с учётом инфраструктуры, поддержки и компетенций команды.

3. Базовая настройка джоб

   **Ответ:** Для базовой job я настраиваю источник кода и credentials, trigger, agent или runner, затем шаги установки зависимостей, сборки и запуска тестов. Также задаю переменные окружения, timeout, правила обработки ошибок, сохранение artifacts и публикацию test reports. Секреты должны храниться в защищённом хранилище CI, а конфигурацию по возможности следует версионировать вместе с кодом.

4. Опыт или работа на проекте с CI.

   **Ответ:** Здесь нужен реальный пример, поэтому ответ можно построить так: «На `[учебном или рабочем проекте]` я использовал `[инструмент]`. Pipeline запускался `[по какому событию]`, выполнял `[реальные этапы]`, а моей ответственностью было `[ваши действия]`». Если коммерческого опыта нет, лучше честно описать самостоятельно настроенный учебный pipeline и показать понимание его этапов.

5. Может рассказать о pipeline на примере пирамиды тестирования и коммита разработчика в монолите.

   **Ответ:** После commit или открытия Pull Request pipeline запускает быстрые проверки: форматирование, static analysis и большой набор Unit tests. Затем собирается монолит, поднимаются необходимые зависимости и выполняются Integration и API tests; после развёртывания в тестовом окружении запускается небольшой набор критичных UI или E2E tests. Чем выше уровень пирамиды, тем обычно меньше тестов и тем позже они запускаются. При ошибке pipeline останавливается, публикует логи и отчёты и не разрешает дальнейшее продвижение сборки.

6. Опыт работы с одним из инструментов CI.

   **Ответ:** Без подтверждённого опыта безопасный шаблон ответа такой: «Я работал с `[Jenkins, TeamCity или другим инструментом]` на уровне `[что именно делали]`: запускал builds, анализировал логи, менял параметры и поддерживал test stage». Затем стоит привести одну реальную задачу и объяснить результат. Если инструмент изучался только самостоятельно, это нужно прямо обозначить и описать учебную конфигурацию.

7. В чем разница между Continuous Delivery и Continuous Deployment.

   **Ответ:** При Continuous Delivery каждое успешное изменение автоматически проходит сборку и проверки и остаётся готовым к выпуску, но production deployment запускается по ручному бизнес-решению. При Continuous Deployment успешное изменение автоматически развёртывается в production без отдельного ручного шага. В обоих случаях нужны надёжные проверки, наблюдаемость и возможность rollback, но Deployment требует особенно высокой зрелости pipeline.

8. Опыт работы и настройка CI на проекте с микросервисной или сервисной архитектурой, может рассказать pipeline.

   **Ответ:** Здесь я бы использовал только реальный пример: `[ваш сервис и ваша ответственность]`. Типичный pipeline определяет изменённый сервис, запускает его Unit, component и contract tests, проверяет код и зависимости, собирает версионированный container image и публикует его в registry. Затем сервис развёртывается в тестовом окружении, где выполняются integration и ограниченные E2E tests; независимые сервисы могут проходить эти этапы параллельно. Важно контролировать совместимость контрактов и не связывать выпуск всех сервисов в один большой pipeline без необходимости.

9. Infrastructure as code в CI.

   **Ответ:** Infrastructure as Code означает, что окружения описываются декларативными файлами и управляются через version control, например с помощью Terraform, Ansible или Kubernetes manifests. В CI можно выполнять format и validation, проверку безопасности и `plan`, а применение изменений защищать approvals и правилами окружения. Такой подход делает инфраструктуру повторяемой и проверяемой, но state и secrets нельзя хранить в открытом виде в репозитории.

10. Опыт настройки CD.

   **Ответ:** Ответ должен опираться на реальный опыт: «Я настраивал `[часть процесса]` для развёртывания в `[окружение]`: добавил `[этапы и проверки]`, настроил `[approval, variables или rollback]`». Если такого опыта не было, можно сказать: «Я самостоятельно CD не настраивал, но понимаю, что после успешного CI artifact продвигается между окружениями с quality gates, управлением конфигурацией, наблюдаемостью и стратегией rollback».

11. Опыт работы с sonar и линтерами.

   **Ответ:** SonarQube или SonarCloud выполняет static analysis и показывает bugs, vulnerabilities, code smells, duplication и coverage, а Quality Gate может остановить pipeline. Линтер проверяет правила языка и стиля быстрее и обычно запускается локально и в CI. Я бы указал только реально использованные инструменты: `[ваш анализатор или линтер]`, а найденные предупреждения оценивал бы по контексту, потому что автоматическое правило может дать false positive.

12. groovy scripts, какие особенности, шелл команды внутри скрипта, try/catch

   **Ответ:** Groovy — динамический язык на JVM, а в Jenkins Pipeline он используется в Jenkinsfile вместе с Pipeline DSL. Shell-команды запускаются через `sh` на Unix-like agents или `bat` и `powershell` на Windows; нужно учитывать quoting, переменные и exit code, потому что ненулевой код обычно завершает stage с ошибкой. `try/catch` позволяет обработать исключение, добавить понятный лог или изменить статус build, а `finally` — выполнить обязательную очистку. Ошибку нельзя просто скрывать: pipeline должен сохранить корректный итоговый статус.
