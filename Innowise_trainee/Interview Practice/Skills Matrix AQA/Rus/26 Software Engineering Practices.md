## Вопросы и ответы

1. OOP:

   **Ответ:** Object-Oriented Programming организует программу как взаимодействие объектов, которые объединяют состояние и поведение. В автоматизации OOP помогает разделять ответственность между тестами, страницами, API-клиентами, моделями данных и утилитами.

2. Понимание основных концепций ООП: классы, объекты, инкапсуляция, наследование, полиморфизм.

   **Ответ:** Класс описывает структуру и поведение, объект является его экземпляром. Инкапсуляция скрывает детали, наследование расширяет базовый тип, а полиморфизм позволяет работать с разными реализациями через общий контракт.

3. Умение объяснить их на простых примерах в контексте языка программирования, используемого для автотестов.

   **Ответ:** Например, класс `LoginPage` хранит локаторы и предоставляет метод `login()`, скрывая действия с элементами. Разные реализации `WebDriver` можно передать через общий интерфейс, поэтому один тест способен работать с разными браузерами.

4. Functional programming:

   **Ответ:** Functional programming строит вычисления из функций и старается уменьшить изменяемое состояние и побочные эффекты. В тестах этот стиль удобен для преобразования коллекций, построения данных и небольших предсказуемых проверок.

5. Очень базовое представление (что это такое, например, неизменяемость данных, чистые функции), если это релевантно для используемого языка/фреймворка.

   **Ответ:** Чистая функция для одинаковых аргументов возвращает одинаковый результат и не меняет внешнее состояние. Неизменяемые данные нельзя изменить после создания, что уменьшает число неожиданных зависимостей в тестах.

6. SOLID:

   **Ответ:** SOLID — пять принципов проектирования объектно-ориентированного кода: SRP, OCP, LSP, ISP и DIP. Они помогают делать компоненты понятными, расширяемыми и слабо связанными.

7. Знакомство с аббревиатурой SOLID. Может назвать некоторые принципы (например, принцип единственной ответственности). Глубокого понимания не требуется.

   **Ответ:** На базовом уровне достаточно объяснить, что Single Responsibility Principle требует одной причины для изменения класса. Например, Page Object не должен одновременно управлять страницей, создавать тестовые данные и формировать отчёт.

8. Code Standards:

   **Ответ:** Code Standards — согласованные правила именования, форматирования, структуры, обработки ошибок и других аспектов кода. Единые правила ускоряют чтение, review и поддержку тестового проекта.

9. Понимание важности следования стандартам кодирования команды/проекта.

   **Ответ:** Общий стиль уменьшает субъективные споры и делает код предсказуемым для всей команды. Новое изменение должно соответствовать правилам репозитория, даже если автор предпочитает другой стиль.

10. Умение писать код, соответствующий базовым требованиям (именование переменных, форматирование).

   **Ответ:** Имена должны отражать назначение, форматирование — быть единым, а методы — оставаться небольшими и понятными. Автоформаттер следует запускать до отправки кода на review.

11. Code Smells:

   **Ответ:** Code smell — признак возможной проблемы в структуре кода, а не обязательно ошибка. Он показывает место, которое стоит изучить и, возможно, улучшить рефакторингом.

12. Понимание, что такое "code smells" на интуитивном уровне (e.g., "длинный метод", "дублирование кода").

   **Ответ:** Длинный метод трудно понимать и тестировать, а дублирование заставляет менять одну логику в нескольких местах. Другие простые признаки — непонятные имена, слишком много параметров и класс с несколькими обязанностями.

13. Refactoring:

   **Ответ:** Refactoring изменяет внутреннюю структуру кода без изменения наблюдаемого поведения. Его цель — повысить читаемость, поддерживаемость и удобство дальнейших изменений.

14. Понимание, что такое рефакторинг и зачем он нужен (улучшение читаемости, упрощение поддержки).

   **Ответ:** Рефакторинг устраняет накопившуюся сложность и дублирование, поэтому код легче понимать и безопаснее развивать. Перед изменением поведение желательно защитить тестами.

15. Умение выполнять простейший рефакторинг под руководством (e.g., переименование переменной, выделение простого участка кода в метод).

   **Ответ:** Сначала выполняют одно небольшое изменение, например Rename или Extract Method, затем компилируют проект и запускают тесты. Маленькие шаги упрощают review и поиск причины возможной ошибки.

16. Code Review Process & Experience:

   **Ответ:** Code review — проверка изменения коллегами до объединения с основной веткой. Она помогает находить дефекты, распространять знания и поддерживать архитектурные и командные стандарты.

17. Понимание цели процесса код-ревью.

   **Ответ:** Цель review — улучшить корректность, понятность, безопасность и сопровождаемость изменения. Это проверка кода и решения, а не оценка личности автора.

18. Умение адекватно воспринимать обратную связь по своему коду.

   **Ответ:** Нужно уточнять непонятные комментарии, обсуждать решение фактами и спокойно исправлять обоснованные замечания. Несогласие лучше объяснять техническими аргументами и примерами.

19. Участие в код-ревью в качестве "ревьюируемого" (автора кода).

   **Ответ:** Автор создаёт небольшой понятный pull request, описывает цель и проверку, самостоятельно просматривает diff и отвечает на каждый комментарий. После исправлений он повторно запускает необходимые проверки.

20. Debugging:

   **Ответ:** Debugging — систематический поиск причины неправильного поведения. Обычно сначала воспроизводят проблему, собирают данные, локализуют участок кода, проверяют гипотезу и подтверждают исправление.

21. Умение использовать базовые инструменты отладки в IDE (точки останова, пошаговое выполнение, просмотр значений переменных) для поиска простых ошибок в своих тестах.

   **Ответ:** Breakpoint останавливает программу, Step Over выполняет текущую строку, Step Into входит в метод, а Watches показывают значения выражений. Так можно проследить состояние теста и найти первую точку, где результат становится неверным.

22. OOP:

   **Ответ:** На среднем уровне OOP применяется осознанно: обязанности распределяются между объектами, зависимости передаются снаружи, а контракты отделяются от реализаций. Наследование не используют только ради повторного использования нескольких строк.

23. Уверенное применение принципов ООП при написании автотестов (e.g., создание Page Object моделей, классов для тестовых данных).

   **Ответ:** Page Object инкапсулирует работу со страницей, API client — HTTP-взаимодействие, а data model — структуру данных. Сам тест описывает сценарий и проверки, не дублируя технические детали.

24. Понимание и умение применять композицию vs наследование, абстрактные классы, интерфейсы.

   **Ответ:** Композиция собирает объект из зависимостей и обычно даёт меньше связанности, чем наследование. Интерфейс задаёт контракт, абстрактный класс может добавить общее состояние и реализацию, а наследование подходит только для настоящего отношения «является» с соблюдением контракта базового типа.

25. Functional programming:

   **Ответ:** На среднем уровне важно применять функциональные элементы там, где они делают код проще. Не следует превращать понятный последовательный тест в сложную цепочку функций только ради стиля.

26. Понимание основных концепций функционального программирования (чистые функции, функции высшего порядка, неизменяемость) и умение применять их элементы там, где это уместно в тестовом коде (e.g., работа с коллекциями через map/filter/reduce, лямбда-выражения).

   **Ответ:** Функция высшего порядка принимает или возвращает функцию, а `map`, `filter` и `reduce` декларативно обрабатывают коллекцию. Чистые преобразования и immutable-модели уменьшают скрытое состояние, особенно при параллельном выполнении тестов.

27. SOLID:

   **Ответ:** На среднем уровне нужно понимать все пять принципов и их компромиссы. SOLID применяют для решения реальной проблемы связанности и изменений, а не для создания лишних интерфейсов и слоёв.

28. Понимание всех принципов SOLID и способность объяснить их применение на примерах из тестового фреймворка.

   **Ответ:** SRP разделяет обязанности; OCP позволяет добавлять браузер без изменения основной логики; LSP требует взаимозаменяемости реализаций; ISP разделяет крупные интерфейсы; DIP заставляет высокоуровневые тестовые сервисы зависеть от абстракций, например `UserRepository`, а не от конкретной БД.

29. Стремление применять принципы SOLID в своем коде.

   **Ответ:** Перед добавлением логики стоит проверить ответственность класса, направление зависимостей и возможность безопасного расширения. При этом решение должно оставаться минимальным и соответствовать текущей сложности проекта.

30. Code Standards:

   **Ответ:** На среднем уровне разработчик не только знает правила, но и постоянно применяет их. Большинство механических требований автоматизируют через formatter, linter, static analysis и CI.

31. Активное следование стандартам кодирования.

   **Ответ:** Код проверяют локально до pull request, новые файлы оформляют в стиле проекта, а исключения из правил объясняют. Стандарты соблюдаются и в production-коде фреймворка, и в самих тестах.

32. Умение использовать линтеры и статические анализаторы для поддержания качества кода.

   **Ответ:** Linter находит нарушения стиля и простые дефекты, static analyzer исследует типы, потоки данных, потенциальные ошибки и уязвимости. Проверки запускают в IDE и CI, а ложные срабатывания подавляют точечно с объяснением.

33. Code Smells:

   **Ответ:** На среднем уровне code smells нужно распознавать в контексте, оценивать их влияние и выбирать безопасный рефакторинг. Не каждый smell требует немедленного изменения.

34. Способность распознавать распространенные "code smells" (e.g., длинные методы, большие классы, чрезмерное использование условных операторов, дублирование кода) и понимать их негативное влияние.

   **Ответ:** Такие признаки повышают сложность, связанность и риск разных вариантов одной логики. Возможные решения — Extract Method/Class, Strategy вместо растущего условного блока и выделение общего поведения без преждевременной абстракции.

35. Refactoring:

   **Ответ:** На среднем уровне рефакторинг выполняют самостоятельно и небольшими проверяемыми шагами. Сначала фиксируют текущее поведение, затем улучшают структуру и после каждого шага запускают проверки.

36. Опыт проведения рефакторинга своего и чужого кода с целью улучшения его структуры, читаемости и поддерживаемости.

   **Ответ:** В чужом коде сначала нужно понять назначение и зависимости, добавить characterization tests при нехватке покрытия и согласовать существенные изменения с владельцами. Pull request лучше разделить на изменение поведения и чистый рефакторинг.

37. Знание основных техник рефакторинга.

   **Ответ:** Основные техники — Rename, Extract/Inline Method, Extract Class, Move Method, Introduce Parameter Object, Encapsulate Field и Replace Conditional with Polymorphism. Технику выбирают под конкретный smell.

38. Code Review Process & Experience:

   **Ответ:** На среднем уровне специалист регулярно выступает и автором, и reviewer. Он проверяет не только стиль, но также требования, тестируемость, ошибки, архитектуру и достаточность тестов.

39. Активное участие в код-ревью как в роли автора, так и в роли ревьюера для кода коллег своего уровня.

   **Ответ:** Автор даёт контекст и быстро обрабатывает замечания, reviewer изучает задачу и проверяет изменение целиком. Неблокирующие предложения нужно отличать от ошибок, без исправления которых merge опасен.

40. Умение предоставлять конструктивную обратную связь.

   **Ответ:** Комментарий должен указывать конкретное место, объяснять риск и по возможности предлагать вариант решения. Лучше писать «этот общий mutable state создаёт race condition» вместо общей оценки «код плохой».

41. Debugging:

   **Ответ:** На среднем уровне отладка включает анализ логов, stack trace, данных, сети и окружения, а не только пошаговый запуск. Важно найти первопричину, а не исправлять последний видимый симптом.

42. Продвинутые навыки отладки: анализ стектрейсов, условные точки останова, отладка многопоточного кода (если применимо), удаленная отладка (базовое представление).

   **Ответ:** Stack trace читают от исходного исключения и его `cause`, conditional breakpoint срабатывает только при нужном состоянии, а thread view помогает увидеть блокировки и гонки. Remote debugging требует debug-порта и защищённого доступа, потому что такой порт нельзя открывать публично.

43. Умение эффективно локализовывать и исправлять ошибки в автотестах и компонентах тестового фреймворка.

   **Ответ:** Сначала определяют слой сбоя: продукт, тест, данные, окружение или инфраструктура. Затем создают минимальное воспроизведение, сравнивают успешный и неуспешный запуск, исправляют первопричину и добавляют проверку от повторения.

44. OOP:

   **Ответ:** На экспертном уровне OOP используется для проектирования границ и стабильных контрактов большого фреймворка. Специалист понимает стоимость абстракций и выбирает паттерн только при наличии соответствующей проблемы.

45. Глубокое понимание и экспертное применение принципов ООП и паттернов проектирования (e.g., Factory, Singleton, Builder, Strategy) при разработке и архитектуре тестовых фреймворков.

   **Ответ:** Factory скрывает выбор реализации, Builder пошагово создаёт сложные данные, Strategy заменяет алгоритм через общий контракт. Singleton допустим для действительно единственного stateless-сервиса, но mutable Singleton создаёт глобальное состояние и проблемы при параллельных тестах.

46. Способность объяснять сложные концепции ООП и их применение в контексте автоматизации тестирования.

   **Ответ:** Объяснение должно связывать принцип с проблемой, примером кода и компромиссом. Например, Dependency Inversion позволяет заменить реальный API client на test double, но лишний интерфейс без альтернативных реализаций может только усложнить код.

47. Functional programming:

   **Ответ:** На экспертном уровне функциональный подход применяется для управления сложностью, состоянием и конкурентностью. Его сочетают с OOP, выбирая подходящую модель для каждой части решения.

48. Глубокое понимание парадигмы функционального программирования.

   **Ответ:** Оно включает referential transparency, композицию функций, immutable data, рекурсию, функции высшего порядка, lazy evaluation и явное представление эффектов. Важно понимать не только синтаксис lambda, но и влияние подхода на дизайн программы.

49. Способность проектировать части тестового фреймворка или писать утилиты в функциональном стиле, если это повышает читаемость, надежность и эффективность.

   **Ответ:** Чистые функции подходят для генерации и преобразования тестовых данных, сравнения моделей и построения запросов. Ввод-вывод, браузер и БД имеют побочные эффекты, поэтому их лучше держать на границах и передавать результаты в чистую логику.

50. Опыт работы с концепциями, такими как каррирование, мемоизация, монады (если релевантно для языка).

   **Ответ:** Каррирование превращает функцию с несколькими аргументами в цепочку функций, мемоизация кеширует результат чистой функции, а monad задаёт способ последовательной композиции вычислений с контекстом. Эти приёмы полезны только тогда, когда язык и команда поддерживают их без потери понятности.

51. SOLID:

   **Ответ:** На экспертном уровне SOLID оценивают на уровне архитектуры и границ модулей. Принципы балансируют с простотой, производительностью и реальными направлениями изменений.

52. Экспертное владение принципами SOLID и способность применять их при проектировании архитектуры сложных тестовых решений.

   **Ответ:** Высокоуровневый orchestration отделяют от Web, API, DB и messaging adapters, контракты делают небольшими, а зависимости направляют внутрь к устойчивой доменной логике. Это позволяет заменять инструменты без переписывания сценариев.

53. Умение выявлять нарушения SOLID в коде и предлагать пути их исправления.

   **Ответ:** Сначала показывают конкретный эффект нарушения: множество причин изменения, зависимость от деталей или невозможность заменить реализацию. Затем предлагают небольшой рефакторинг, например разделение интерфейса, внедрение зависимости или выделение Strategy, и оценивают цену миграции.

54. Code Standards:

   **Ответ:** На экспертном уровне standards становятся частью инженерного процесса. Они должны быть понятными, версионируемыми, автоматически проверяемыми и пересматриваться при изменении языка, инструментов или архитектуры.

55. Участие в разработке и обновлении стандартов кодирования для команды/проекта.

   **Ответ:** Правила формируют вместе с командой на основе повторяющихся проблем и практик экосистемы. Для каждого важного правила описывают мотивацию, пример, область действия и способ автоматической проверки.

56. Настройка инструментов статического анализа и линтинга, обеспечение их соблюдения.

   **Ответ:** Конфигурацию хранят в репозитории, одинаково запускают локально и в CI, а нарушение обязательных правил блокирует merge. Baseline для старого долга вводят осознанно, чтобы новые изменения не увеличивали его.

57. Code Smells:

   **Ответ:** На экспертном уровне оценивают smells внутри архитектуры и истории изменений. Один и тот же признак может быть приемлемым локально, но опасным в часто меняющемся центральном компоненте.

58. Способность выявлять не только очевидные, но и более тонкие "code smells" и анти-паттерны.

   **Ответ:** Примеры — shotgun surgery, feature envy, temporal coupling, speculative generality, inappropriate intimacy, flaky test logic и общий mutable state. Их замечают по структуре зависимостей, частым совместным изменениям и причинам дефектов.

59. Понимание глубинных причин их возникновения и последствий.

   **Ответ:** Причинами бывают неясные границы ответственности, давление сроков, неверная абстракция и архитектура, не соответствующая развитию продукта. Последствия — высокая связанность, медленные изменения, дефекты, нестабильные тесты и сложное обучение команды.

60. Refactoring:

   **Ответ:** Сложный refactoring требует плана миграции, наблюдаемости и возможности отката. Его делят на этапы так, чтобы система оставалась рабочей и команда могла продолжать разработку.

61. Опыт проведения сложного рефакторинга, затрагивающего архитектуру тестового фреймворка.

   **Ответ:** Сначала описывают целевую архитектуру и зависимости, добавляют защитные тесты, затем постепенно переводят потребителей на новые контракты. Старый путь удаляют только после проверки CI, параллельных запусков, отчётов и основных сценариев.

62. Способность принимать решения о необходимости рефакторинга и планировать его проведение.

   **Ответ:** Сравнивают стоимость текущей проблемы, частоту изменений, риск дефектов и цену рефакторинга. План задаёт scope, этапы, владельцев, проверки, метрики успеха, совместимость и rollback.

63. Владение широким спектром техник рефакторинга.

   **Ответ:** Кроме локальных Extract/Move/Rename, специалист применяет Branch by Abstraction, Strangler, Parallel Change и поэтапное изменение контрактов. Выбор техники зависит от размера системы и необходимости сохранять совместимость.

64. Code Review Process & Experience:

   **Ответ:** На экспертном уровне review является управляемой системой качества, а не только проверкой отдельных pull requests. Важны понятные правила, разумное время реакции, распределение знаний и автоматизация механических проверок.

65. Лидирующая роль в процессе код-ревью: ревью кода любой сложности, менторство других участников, разрешение спорных моментов.

   **Ответ:** Лидер помогает увидеть архитектурные риски, задаёт вопросы, объясняет решения и вовлекает нужных экспертов. Спор разрешают через требования, данные, proof of concept или зафиксированное командное решение, а не через должность участника.

66. Установление и поддержание высоких стандартов качества кода через код-ревью.

   **Ответ:** Definition of Done включает тесты, анализ, документацию и обязательные approvals для рискованных областей. При этом review не заменяет CI: форматирование, lint, compilation и повторяемые проверки выполняются автоматически.

67. Debugging:

   **Ответ:** Экспертная отладка объединяет логи, traces, metrics, dumps, profiler и знание архитектуры. Работа строится вокруг проверяемых гипотез и временной линии события.

68. Экспертные навыки отладки сложных проблем, включая анализ производительности, проблем с памятью (memory leaks, GC issues), многопоточных проблем в тестовых фреймворках и интеграциях.

   **Ответ:** Для производительности используют CPU profile и traces, для памяти — heap dump и анализ удерживающих ссылок, для GC — логи и паузы, для concurrency — thread dump, lock contention и race detection. Данные собирают в момент проблемы и сравнивают с нормальным запуском.

69. Умение использовать продвинутые инструменты отладки и профилирования.

   **Ответ:** В зависимости от стека применяют profiler IDE, Java Flight Recorder, VisualVM, dotnet-trace/dump, browser DevTools, distributed tracing и системные инструменты. Инструмент выбирают по типу гипотезы, а не запускают всё сразу.

70. Способность быстро находить корень проблемы даже в незнакомом коде.

   **Ответ:** Сначала строят карту входов, выходов и зависимостей, воспроизводят сбой и находят первый неверный факт в цепочке. Логи, tests, call hierarchy и `git history` помогают сузить область, после чего гипотезу подтверждают минимальным экспериментом.
