## Вопросы и ответы

1. Брокер сообщений. Назначение. Как используется

   **Ответ:** Message Broker принимает сообщения от producer, временно хранит или маршрутизирует их и передаёт consumer. Он разделяет сервисы по времени и зависимости: отправитель может не знать, кто и когда обработает сообщение. Брокеры используют для асинхронной обработки, интеграции сервисов, очередей задач и событий.

2. Примеры брокеров сообщений

   **Ответ:** Примеры — Apache Kafka, RabbitMQ, ActiveMQ Artemis, Amazon SQS, Azure Service Bus, Google Cloud Pub/Sub и NATS. Выбор зависит от требуемой модели доставки, порядка сообщений, пропускной способности, хранения и маршрутизации.

3. Инструменты сбора логов

   **Ответ:** Логи собирают Fluent Bit, Fluentd, Logstash, Filebeat, Vector или агенты облачных платформ. Их отправляют в Elasticsearch/OpenSearch, Loki, Splunk или облачное хранилище, а для поиска и визуализации используют Kibana или Grafana.

4. Что такое topic

   **Ответ:** Topic — именованный поток или категория сообщений. Producer записывает события в topic, а consumer подписывается и читает их; в Kafka один topic может иметь несколько producer, consumer и partitions.

5. Что такое партитишен

   **Ответ:** Partition — часть topic в Kafka, представляющая упорядоченный журнал сообщений. Partitions распределяются между brokers и позволяют масштабировать запись и чтение; строгий порядок гарантируется только внутри одной partition.

6. Отличия kafka и RabbitMQ

   **Ответ:** Kafka — распределённый event log: сообщения хранятся заданное время, consumer управляет offset и может перечитать события. RabbitMQ — классический Message Broker: exchanges маршрутизируют сообщения в queues, а после подтверждённой обработки сообщение обычно удаляется. Kafka часто выбирают для большого потока событий и повторного чтения, RabbitMQ — для очередей задач и гибкой маршрутизации.

7. Инструменты мониторинга сети

   **Ответ:** Для анализа пакетов применяют Wireshark и tcpdump, для проверки соединений — ping, traceroute, curl, nslookup/dig, netstat или ss. Для постоянного мониторинга используют Prometheus exporters, Zabbix, Grafana, облачные Network Monitoring и APM-инструменты.

8. Подключение к Брокерам сообщений

   **Ответ:** Клиенту задают адреса brokers, порт, протокол, credentials, TLS-сертификаты и параметры timeout/retry. Для Kafka обычно указывают `bootstrap.servers`, а для RabbitMQ — AMQP URI; доступ сначала проверяют CLI или административным UI.

9. Чтение и запись сообщений

   **Ответ:** Producer сериализует сообщение и отправляет его в topic или exchange, при необходимости задавая key и headers. Consumer получает и десериализует сообщение, проверяет данные, обрабатывает его и подтверждает результат через acknowledgement или фиксирует offset.

10. Из каких компонентов состоит kafka

   **Ответ:** Kafka cluster состоит из brokers, topics и partitions с replicas. Producer записывает события, consumer читает их, consumer groups распределяют partitions, а controllers управляют метаданными и состоянием кластера; в современных версиях для этого используется KRaft.

11. kafka schema

   **Ответ:** Schema описывает структуру и типы полей сообщения, например в Avro, JSON Schema или Protobuf. Обычно schema хранят в отдельном Schema Registry, а producer и consumer проверяют совместимость версий, чтобы изменение формата не сломало обмен данными. Сам Kafka не требует schema для каждого сообщения.
