## Вопросы и ответы

1. Рассказать про инструмент, с которым работал(в общих чертах что за он и что умеет)

   **Ответ:** Здесь нужно назвать реальный инструмент и свою задачу: «Я использовал `[инструмент]` для `[вид тестирования]`. Он умеет `[основные возможности]`, а на проекте я применял его для `[реальный пример]`». Например, Selenium WebDriver управляет браузером через WebDriver protocol, но для полноценного проекта ему дополнительно нужны test runner, assertions, отчётность и собственная архитектура framework.

2. Рассказать из каких блоков состоит тест

   **Ответ:** Тест обычно следует структуре Arrange–Act–Assert: подготовить данные и состояние, выполнить действие, затем проверить результат. В автоматизации также могут быть setup и teardown для окружения, а общие технические действия выносятся в helpers, Page Objects или API clients. Сам тест должен ясно показывать бизнес-сценарий и ожидаемый результат, не скрывая важную логику в общих методах.

3. Какие фреймворки еще слышал

   **Ответ:** Для Java я знаю JUnit и TestNG как test frameworks, Selenium и Selenide для web automation, REST Assured для API и Cucumber для BDD. В других стеках распространены pytest, NUnit, Playwright и Cypress. На собеседовании важно различать назначение инструментов: например, JUnit управляет жизненным циклом тестов, а Selenium предоставляет API для работы с браузером.

4. Как запустить/пропустить отдельный тест

   **Ответ:** Способ зависит от test framework и build tool. В JUnit 5 отдельный тест можно запустить из IDE или через Maven, указав класс и метод, а временно пропустить — с помощью `@Disabled`; в TestNG для этого можно использовать настройки suite или `@Test(enabled = false)`. Постоянно скрывать падающий тест нельзя: причину пропуска и условие возврата следует зафиксировать.

5. Критерии выбора фреймворка

   **Ответ:** Я учитываю тип приложения и уровень тестирования, язык и стек команды, поддержку нужных браузеров или протоколов, возможности параллельного запуска, отчётности и CI. Также важны зрелость сообщества, документация, частота обновлений, лицензия и стоимость поддержки. Перед окончательным выбором полезно сделать небольшой proof of concept на критичном сценарии.

6. Какие есть хуки(before, after и т.п. и в чем разница)

   **Ответ:** Hooks выполняют подготовку и очистку до или после suite, class, group либо каждого test method. Например, в JUnit 5 есть `@BeforeAll`, `@AfterAll`, `@BeforeEach` и `@AfterEach`: первые работают один раз для класса, вторые — для каждого теста. Выбор scope важен, потому что общие изменяемые данные между тестами создают зависимости, особенно при параллельном запуске.

7. Рассказать про конфигурационный файл своего фреймворка

   **Ответ:** В ответе я бы показал реальный файл: `[название файла]`, и объяснил его параметры. Обычно конфигурация хранит URL окружения, browser, timeout, режим параллельного запуска, теги и настройки отчётов; для TestNG suite также может задаваться в `testng.xml`. Секреты не должны попадать в репозиторий: их передают через environment variables или защищённое хранилище CI.

8. Запуск по тегам?

   **Ответ:** Да, тестам можно назначать категории и запускать выбранные группы. В JUnit 5 используется `@Tag`, в TestNG — `groups`, а в Cucumber — tags сценариев; build tool или конфигурация runner задаёт include и exclude filters. Например, так можно отдельно запускать Smoke на Pull Request и более полный Regression testing по расписанию.

9. Можно ли запускать тесты параллельно и как?

   **Ответ:** Да, если test runner поддерживает parallel execution и тесты изолированы. Параллелизм настраивается по методам, классам, suites или на уровне CI agents, но каждому потоку нужны независимые driver instances, данные и состояние. Общие mutable objects, один пользователь или одно окружение могут создавать race conditions, поэтому сначала нужно обеспечить thread safety и контролировать нагрузку на систему.

10. Как подключить репортер

   **Ответ:** Обычно нужно добавить dependency репортера, зарегистрировать listener, extension или plugin и указать каталог для результатов. После выполнения тестов генерируется отчёт, например Allure Report, а CI сохраняет и публикует его как artifact. В отчёт полезно добавлять шаги, логи, screenshots и данные окружения, но нельзя включать passwords, tokens и персональные данные.

11. Хранение и работа с данными

   **Ответ:** Статические данные можно хранить в читаемых файлах ресурсов, а динамические создавать через builders, factories или API перед тестом. Конфигурацию окружения лучше отделять от тестовых данных, а secrets передавать через защищённые переменные. Тест должен создавать известное начальное состояние и по возможности очищать свои данные, чтобы прогоны не зависели друг от друга.

12. Интеграция с системами CI

   **Ответ:** Framework должен запускаться одной воспроизводимой командой без IDE и возвращать корректный exit code. В pipeline настраиваются dependencies, environment variables, test command, параллелизм, timeout, сохранение logs и публикация reports. Для разных этапов можно использовать теги: быстрый Smoke на Pull Request, расширенный набор после deployment и Regression testing по расписанию.

13. Интеграция с тест-кейс меджмент системами

   **Ответ:** Автотест связывают с test case через стабильный идентификатор, annotation или mapping file, а результаты отправляют в систему через API, plugin или готовый adapter. Обычно перед прогоном создаётся test run, после чего передаются статус, длительность, ошибка и ссылка на CI artifacts. Интеграция не должна превращать название test method в единственный идентификатор, иначе обычный refactoring разорвёт связь.

14. Расширение базовых возможностей фреймворка

   **Ответ:** Возможности расширяют через extension API, listeners, hooks, custom annotations, plugins и собственные adapters. Например, listener может автоматически делать screenshot при падении, а extension — создавать и закрывать browser driver для каждого теста. Расширение должно решать повторяющуюся задачу, иметь понятный интерфейс и не скрывать ошибки или жизненный цикл теста.

15. отличия JUnit и TestNG для автоматизации

   **Ответ:** JUnit 5 состоит из Platform, Jupiter и Vintage, поддерживает extensions, tags, parameterized и nested tests и широко используется в Java-проектах. TestNG изначально предоставляет groups, dependencies между тестами, priorities, DataProvider, XML suites и гибкую настройку параллельного выполнения. Оба подходят для автоматизации; выбор зависит от существующего стека и требований, а зависимости и priorities TestNG нужно применять осторожно, чтобы тесты не стали связанными по порядку.

16. генерация тестовых данных

   **Ответ:** Тестовые данные можно создавать программно через builders и factories, библиотеками вроде Faker, генераторами случайных значений или property-based testing. Генератор должен соблюдать бизнес-ограничения и уметь создавать граничные и негативные значения. Для воспроизводимости случайного падения нужно сохранять seed и фактические данные, а production и конфиденциальную информацию использовать нельзя без разрешённой защиты и обезличивания.
