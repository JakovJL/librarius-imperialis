## Вопросы и ответы

1. Система контроля версий (что это такое и для чего нужна).

   **Ответ:** Система контроля версий хранит историю изменений файлов и позволяет нескольким людям безопасно работать над одним проектом. С её помощью можно создавать ветки, сравнивать версии, объединять изменения и возвращаться к известному состоянию. Git — распределённая система, а SVN и TFVC используют централизованную модель.

2. Как склонировать проект.

   **Ответ:** Git-репозиторий клонируется командой `git clone <URL>`, которая загружает историю, создаёт локальную копию и настраивает remote `origin`. При необходимости можно сразу указать папку: `git clone <URL> <folder>`. Для приватного репозитория предварительно настраиваются доступ по SSH или Authentication через HTTPS.

3. Рассказать о командах git pull, git push, git commit, при условии что работал с git

   **Ответ:** `git commit` сохраняет подготовленные через `git add` изменения в локальной истории. `git push` отправляет локальные коммиты в удалённый репозиторий, а `git pull` получает удалённые изменения и интегрирует их в текущую ветку через merge или rebase в зависимости от настройки. Перед этими командами я обычно проверяю текущую ветку и `git status`, чтобы не отправить или не объединить лишние изменения.

4. git branch

   **Ответ:** `git branch` показывает локальные ветки, `git branch <name>` создаёт новую, а `git branch -d <name>` безопасно удаляет уже объединённую ветку. Для создания ветки и немедленного перехода на неё используется `git switch -c <name>` или старый вариант `git checkout -b <name>`. Команда сама по себе не переносит ветку на remote.

5. Как сделать локальную ветвь удаленной

   **Ответ:** Локальную ветку публикуют командой `git push -u origin <branch>`. Флаг `-u` связывает локальную ветку с удалённой upstream branch, поэтому дальше обычно достаточно `git push` и `git pull`. Перед публикацией я проверяю название ветки и remote через `git remote -v`.

6. svn co, svn up, если работал

   **Ответ:** В SVN команда `svn checkout <URL>` или сокращённо `svn co <URL>` создаёт working copy из репозитория. `svn update` или `svn up` получает актуальные изменения и объединяет их с локальной рабочей копией. Если практического опыта с SVN нет, на собеседовании лучше прямо сказать об этом и затем объяснить назначение известных команд.

7. tfs, если работал

   **Ответ:** TFS, который сейчас развивается как Azure DevOps Server, может хранить код в Git или централизованной системе TFVC. В TFVC используются workspace, check-out при соответствующей настройке, pending changes, check-in, shelvesets и команды получения последней версии. Если реального опыта нет, я бы честно сказал: «С TFS я не работал, но понимаю отличие TFVC от Git и готов освоить workflow проекта».

8. git config (задание имени и email)

   **Ответ:** Имя и email для текущего пользователя задаются командами `git config --global user.name "Name"` и `git config --global user.email "name@example.com"`. Без `--global` настройка применяется только к текущему репозиторию и переопределяет глобальное значение. Проверить результат можно через `git config user.name` и `git config user.email`.

9. Флаг --hard (при fetch reset) или как перезаписать локальные изменения серверными

   **Ответ:** Сначала выполняют `git fetch origin`, затем для полного выравнивания текущей ветки с remote можно использовать `git reset --hard origin/main`. Команда безвозвратно убирает локальные изменения tracked files и локальные коммиты, которых нет в указанной remote branch, поэтому перед ней нужно проверить ветку и при необходимости сделать backup branch или `git stash`. Untracked files эта команда обычно не удаляет.

10. Как перенести изменени из одной ветку в другую (3 способа)

   **Ответ:** Первый способ — `merge`, который переносит всю линию изменений и при необходимости создаёт merge commit. Второй — `rebase`, который повторно применяет коммиты текущей ветки поверх другой основы и делает историю линейной. Третий — `cherry-pick`, который переносит выбранные коммиты; для незакоммиченных изменений также можно временно использовать `stash`, но это не основной способ переноса истории ветки.

11. git rebase (что это такое и как работает, как отменить rebase)

   **Ответ:** `git rebase <base>` меняет основу текущей ветки, последовательно повторно применяя её коммиты поверх `<base>`. При конфликте нужно исправить файлы, выполнить `git add` и `git rebase --continue`; до завершения процесс отменяется через `git rebase --abort`. После завершённого локального rebase прежнее состояние можно найти через `git reflog` и восстановить с помощью `git reset`, а переписывать уже общую опубликованную историю без согласования опасно.

12. git reset

   **Ответ:** `git reset` перемещает указатель текущей ветки на выбранный commit. Режим `--soft` сохраняет изменения в staging area, стандартный `--mixed` оставляет их только в working tree, а `--hard` удаляет изменения tracked files. Для общей опубликованной истории обычно безопаснее `git revert`, потому что он создаёт новый обратный commit и не переписывает историю.

13. git stash .ignore

   **Ответ:** `git stash` временно сохраняет незакоммиченные изменения tracked files и очищает working tree; `git stash pop` применяет их и удаляет запись из stash, а `git stash apply` оставляет запись. Файл `.gitignore` задаёт шаблоны файлов, которые Git не должен начинать отслеживать, например build artifacts или локальные настройки. Если файл уже tracked, добавление в `.gitignore` само по себе не перестанет его отслеживать.

14. git log (как посмотреть историю изменений)

   **Ответ:** `git log` показывает историю коммитов текущей ветки с автором, датой и сообщением. Для краткого дерева веток удобно использовать `git log --oneline --graph --decorate --all`, а `git log -- <file>` показывает историю конкретного файла. Дополнительные фильтры позволяют искать по автору, дате или тексту сообщения.

15. мержи, реверты в svn, tfs, если не работали с git

   **Ответ:** В SVN merge применяет различия из выбранной ветки или диапазона ревизий к working copy, после чего результат проверяется и фиксируется обычным commit. Обратное слияние ревизии позволяет отменить изменение, сохранив новую запись в истории. В TFVC аналогичные задачи выполняются через merge между branches и rollback изменений; точные шаги зависят от клиента и настроек проекта.

16. создание, ревью и мерж pull requests

   **Ответ:** Сначала я создаю отдельную ветку, делаю небольшие понятные commits, публикую ветку и открываю Pull Request в целевую branch. В описании указываю цель, основные изменения, способ проверки и связанные задачи; затем отвечаю на замечания и обновляю код. После успешных проверок и необходимых approvals PR объединяется выбранной стратегией — merge, squash или rebase — согласно правилам репозитория.

17. git cherry-pick (как выбрать коммиты)

   **Ответ:** `git cherry-pick <commit>` применяет изменение выбранного commit поверх текущей ветки и создаёт новый commit с другим hash. Несколько commits можно перечислить явно или указать корректный диапазон. При конфликте нужно исправить файлы и продолжить через `git cherry-pick --continue`, а отменить незавершённую операцию можно командой `git cherry-pick --abort`.

18. git submodules (как добавить подмодуль, обновить, удалить)

   **Ответ:** Подмодуль добавляют командой `git submodule add <URL> <path>` и фиксируют изменения `.gitmodules` и ссылки на конкретный commit. После клонирования подмодули инициализируют и обновляют через `git submodule update --init --recursive`; для загрузки удалённых обновлений используется `git submodule update --remote`. При удалении сначала выполняют `git submodule deinit -f <path>`, затем `git rm <path>` и фиксируют изменения, проверяя, что запись исчезла из `.gitmodules`.

19. git hooks

   **Ответ:** Git hooks — это скрипты, которые запускаются на событиях Git, например перед commit, push или после получения изменений сервером. Их используют для форматирования, статического анализа, проверки сообщения commit и других быстрых контролей. Локальные hooks из `.git/hooks` не передаются обычным clone, поэтому общие проверки лучше хранить в репозитории через специальный инструмент и дублировать критичные правила в CI.
